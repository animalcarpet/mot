Why a pattern library?

The pattern library is a reference tool for designers and developers that contains all the design elements that appear multiple times across the application. It documents the code, look, and behaviour of the components and is important for a number of reasons:

  • it increases the understanding of the application through consistent naming and pattern reuse
  • reduces cognative load on users through pattern reuse
  • it gives designers and developers the building blocks to quickly build complex interactions
  • it facilitates rapid prototyping of new ideas
  • improves the consistency across the application

How can a developer use it?

The pattern library acts as a functional specification for developers. Each pattern will detail the html code, the expected behaviour on interaction, and when and where the pattern should be used.

Developers can match the patterns found in UX Pin to those in the pattern library and lift the corresponding code to build the component. This ensures that shared componets are built in a consistent manor and in turn improving the user experience.

How can a BA/PO use it?

The main focus for the BA/PO is understanding when and where to use a pattern. By referencing the pattern library BA/PO's can look for existing patterns that match the needs for their story.

Pattern names play an important part as they create a shared vocabulary across teams. This will result in improved communication of ideas as different regions of the site can be refered too and everyone will have a shared understanding of what that area might look and behave like.

Changes to the pattern library

You can request a change to a pattern or a request a new pattern if you feel the requirements for the story are not met. New patterns come at a cost as they require some design, development, and add an additional load on users to interpret another pattern.

Patterns will only be created or extended when there is a sound business case for doing so. Before a change or addition can be made you must provide evidence to show why an existing pattern cannot be used. You must also state the business/user needs for the change or additional pattern. If a developer is requesting the change they must get sign off from both the BA and PO, if a BA is requesting the change then sign off must be recieved from the PO.

Once sign off has been approved a UX team member will come up with a design solution and ensure all parties are happy with the new pattern. The pattern will then be presented to the wider UX team to ensure consistency with the application before being approved for entry into the pattern library.
